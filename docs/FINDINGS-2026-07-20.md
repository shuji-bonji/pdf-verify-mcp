# evaluate_policy の穴 2 件（2026-07-20・④ 実連携で発見）

出典: `Document-Note/mcps/PDFfamily/specs/11-kickoff-measurements.md` §1。
PDF family の ④（Skill の見直し + 実連携）で pdf-trust Skill を 1 本通した際に出たもの。

**どちらも「判定が間違っている」のではなく「判定に必要な事実が表に出ない／ルールが素通りする」**。
結果として **Skill（LLM）が不足分を自力で補いたくなる** = 委譲の設計が壊れる方向の圧力になる。
pdf-trust は「ジャッジはコード、ナラティブは LLM」を中核原則に置いているため、
engine 側で塞ぐのが筋（Skill 側で判定を変えるのは禁止事項）。

検体・再現:

| 検体 | 用途 |
|---|---|
| `pdfnative-audit/out4/signed-then-annotated.pdf` | 非認証署名 + 署名後に 574 バイトの増分更新 |
| `tests/fixtures/generated/certified-p1-modified.pdf` | 認証署名（DocMDP P=1）+ 違反あり（**負の対照**） |

どちらも `profile: contract` / `trust_anchors` なし / `check_revocation: embedded`。

---

## V-A1 🟠 非認証署名では「署名後の変更」が facts にも advisories にも出ない

### 観測

同じファイルに対して:

| ツール | 出力 |
|---|---|
| `verify_integrity` | `bytesAfterSignedRange: 574` / `lastSignatureCoversFile: false` / `incrementalUpdateCount: 2` |
| `evaluate_policy` の `facts` | **該当項目なし**（`signatureCount` / `signatures` / `revisionCount` / `certification` / `hasDss` / `padesLevels` / `conformance` のみ） |
| `evaluate_policy` の `advisories` | **空** |

`verdict` は `use_with_caution`（trust 未評価 + 失効不明）で、それ自体は妥当。
問題は**「署名後に何かが足された」という事実が呼び出し側に一切渡らない**こと。

### 原因（コード上）

`services/policy-engine.ts` は `facts.integrity` を内部では参照している:

- L184-185: `f.integrity.certification?.violatedByLaterChanges` → `POL-REVIEW-DOCMDP-VIOLATION`
- L305: `facts.integrity.certification?.laterChangesAppearLtvOnly` → advisory

**どちらも `certification?.` の下にある。** つまり認証署名（DocMDP）のときだけ、
後続変更が表に出る。**非認証の通常署名では経路が存在しない。**

負の対照でこれは確認済み: `certified-p1-modified.pdf` では
`certification.violatedByLaterChanges: true` が facts に出て
`POL-REVIEW-DOCMDP-VIOLATION` が発火し `human_review_required` に落ちる。
ルールエンジン自体は機能している。**欠けているのは非認証パスだけ。**

### なぜ困るか

`references/contract.md` の必須チェック 3 は
「`verify_integrity` — 署名後の増分更新を確認。連署（複数署名）は正常」を要求する。
`evaluate_policy` がこれを扱わないため、Skill は `verify_integrity` を別途呼んで
レポートに載せることになる。今回は規約どおり advisories 相当に留めたが、
**「判定に効くべきでは」と Skill 側が考え始める余地**を残している。

増分更新は合法（連署・DSS 追加）なので `verdict` を動かすべきではない。
**だからこそ advisory として出すのが正しい形。**

### 提案

1. 返す `facts` に integrity の要約を足す
   （`bytesAfterSignedRange` / `lastSignatureCoversFile` / `incrementalUpdateCount`）
2. 非認証署名で後続変更があるとき advisory を 1 本足す。
   verdict は変えない。文面は「増分更新は PDF で合法だが、追加された内容は確認すべき」
   という `verify_integrity` の `notes` と揃える

## V-A2 🟠 非 PAdES 署名で PAdES advisory が沈黙する（`null` の素通り）

### 観測

同じ `profile: contract`（`recommendedMinPadesLevel: B-T`）で:

| 検体 | `padesLevels` | PAdES advisory |
|---|---|---|
| `certified-p1-modified.pdf` | `B-B` | ✅ 発火 |
| `signed-then-annotated.pdf` | **`null`** | ❌ **沈黙** |

`null` は `detect_pades_level` が
`"Legacy ISO 32000-1 signature (adbe.pkcs7.detached) — not a PAdES baseline signature."`
と返すケース。**長期検証性は B-B より悪い**（タイムスタンプも DSS も無い）のに、
B-B では出る警告が出ない。

### 原因（コード上）

`services/policy-engine.ts` L291:

```ts
const below = facts.pades.filter((p) => p.level !== null && order.indexOf(p.level) < min);
```

`p.level !== null` のガードが、**非 PAdES 署名を比較対象から外している**。
`order.indexOf(null)` が `-1` になって誤判定するのを避ける意図だと思われるが、
結果として「そもそも PAdES ですらない」= 最も警告すべきケースが漏れている。

### 提案

`level === null` を「どの推奨水準も満たさない」として扱い、別文面の advisory を出す。
B-B 用の文面（「LTV 化を検討」）は PAdES 前提なので、非 PAdES には
「レガシー署名（`adbe.pkcs7.detached`）であり PAdES ベースラインを満たさない。
長期保存が要件なら PAdES への移行が要る」といった別メッセージが適切。

verdict は変えない（現状どおり advisory 止まり）。

---

## 共通の注意

`profile.recommendedMinPadesLevel` を持つのは contract 以外にもある。
修正時は**全プロファイルで `level: null` の検体を通す**こと。
今回の発見は、負の対照（B-B の検体）と並べて初めて「片方だけ黙っている」と分かった。
**単独の検体では「advisory が無いのは正常」と読めてしまう。**
